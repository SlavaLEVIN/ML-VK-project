# Улучшение изображений в браузере (ML-модуль на TensorFlow.js)

Учебный проект: клиентское веб-приложение, которое улучшает фотографии
(яркость, контрастность, насыщенность) прямо в браузере пользователя,
без отправки изображения на сервер.

## Как запустить

Файлы используют Web Worker и `fetch`-подобные механизмы браузера, поэтому
открыть `index.html` двойным кликом (протокол `file://`) не получится -
нужно раздать папку через локальный веб-сервер. Проще всего:

```bash
cd image-enhancer
python3 -m http.server 8000
```

и затем открыть в браузере `http://localhost:8000`.

Интернет-соединение при запуске нужно — библиотека TensorFlow.js и
конвертер HEIC подключаются с CDN (`jsdelivr`), чтобы не раздувать
размер собственного кода проекта.

## Как пользоваться

1. Нажать «Выбрать изображение» и загрузить JPG/PNG/HEIC/BMP.
2. Нажать «Обработать» — запускается задача (`submitTask`).
3. Прогресс обновляется через событие `taskstatuschange` и прогресс-бар.
4. По завершении справа появится результат и ссылка «Скачать результат».
5. Кнопка «Отменить» в любой момент прерывает обработку (`abortTask`).

## Как это работает

Обработка разбита на два этапа, как и указано в ТЗ:

1. **ИИ-модуль подбирает параметры.** В `worker.js` изображение
   уменьшается до 128×128, и с помощью TensorFlow.js вычисляются три
   признака: средняя яркость, контраст (СКО яркости) и средняя
   насыщенность, а также отсортированный массив значений яркости (для
   расчёта перцентилей). Признаки подаются на вход небольшой нейросети
   (`tf.layers.dense` + sigmoid), которая возвращает два числа 0..1:
   насколько агрессивно растягивать контраст и насколько усилить цвет.

2. **Вспомогательный алгоритм применяет параметры.** Используется
   приём **auto-levels (растяжение контраста по перцентилям яркости)**:
   находятся "тёмная" и "светлая" границы (например, 1-й и 99-й
   перцентиль), и весь диапазон яркости линейно растягивается в 0..255
   — одним и тем же коэффициентом для R, G и B, чтобы не сдвигать
   цветовой баланс. Затем насыщенность **увеличивается** (никогда не
   уменьшается) относительно яркости пикселя:
   - контраст/яркость: `p = p * scale + offset` (scale/offset — из перцентилей)
   - насыщенность: `p = luminance + (p - luminance) * saturationBoost`, где `saturationBoost >= 1.0`

   Именно такой порядок - ключевое отличие от первой версии алгоритма:
   раньше контраст и насыщенность "стягивались" к неким средним целевым
   значениям, из-за чего уже нормальные/контрастные фото становились
   более тусклыми (модель их "усредняла"). Auto-levels по перцентилям —
   адаптивный приём: он растягивает именно тот диапазон яркости, который
   есть в конкретном фото, поэтому результат всегда контрастнее и ярче
   исходника, а не тусклее.

   Между растяжением контраста и усилением насыщенности добавлен
   промежуточный шаг — **лёгкая коррекция баланса белого** (метод
   "серого мира", применяется лишь частично, на 40%). Без него усиление
   насыщенности не только делает цвета ярче, но и **усиливает уже
   существующий цветовой перекос** (например, от тёплого лампового
   света) — фото становится избыточно тёплым/жёлтым. Частичная
   коррекция баланса белого сглаживает перекос перед усилением цвета,
   не убирая естественное освещение полностью.

   Цикл выполняется частями с периодическими `await` и проверкой флага
   отмены — это обеспечивает асинхронность (интерфейс не блокируется)
   и поддержку прерывания задачи.

### Важно про "обучение" модели

Веса нейросети в `worker.js` заданы **аналитически**, а не
получены обучением на датасете — сеть реализует понятное правило «чем
слабее исходный контраст/насыщенность, тем сильнее коррекция». Это
осознанное упрощение проекта: настоящая модель — это реальный объект `tf.js`, 
который делает `predict()`, но её веса не подбирались градиентным спуском.

## Соответствие требованиям ТЗ

| Требование | Как реализовано |
|---|---|
| Работа во всех современных браузерах | Chrome/Edge/Firefox/Safari — стандартные API (Worker, Canvas, createImageBitmap) |
| Объём кода до 10 МБ | Собственный код проекта — не больше сотни КБ; тяжёлые библиотеки (TF.js, heic2any) подключаются с CDN, а не входят в поставку |
| Обработка изображений до 15 Мпк | Пиксельный цикл работает с любым размером; для очень больших сторон изображение можно предварительно уменьшать в `_decodeFile`, если потребуется — сейчас ограничение не задано жёстко |
| Максимум 30 с / в среднем 5 с | На современном ноутбуке обработка фото ~12–15 Мпк занимает единицы секунд; точное время зависит от железа |
| JPG/PNG/HEIC/BMP | JPG/PNG/BMP — через `createImageBitmap`; HEIC/HEIF — конвертация в JPEG библиотекой `heic2any` перед обработкой |
| Асинхронный режим, без блокировки | Вся тяжёлая работа — в Web Worker, пиксельный цикл дополнительно "отдаёт" event loop каждые ~2–3% прогресса |
| API: submitTask/getStatus/abortTask/getResult | Реализовано в `app.js` (класс `ImageEnhancerAPI`) |
| Событие изменения статуса | `CustomEvent('taskstatuschange', {detail:{taskId,status,progress}})` |
